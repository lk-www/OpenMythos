# Transformer 基礎概念まとめ

---

## 1. Transformer とは

**「トークン同士の関係を並列に計算する」** ニューラルネットワークのアーキテクチャ。2017年の論文 *"Attention is All You Need"* で提案された。

### それ以前の問題（RNN）

```
x1 → h1 → x2 → h2 → x3 → h3 → ...
```

- **逐次処理**のため並列化できない
- 長い文になると**遠くのトークンを忘れる**
- 学習が遅い

### Transformer が解決したこと

```
x1 ─┐
x2 ─┼─ Attention ─→ 全トークンを同時に相互参照
x3 ─┘
```

- **並列処理**できる（GPU と相性が良い）
- どんなに離れたトークン間でも**直接参照**できる
- 長文でも情報が失われにくい

### 全体構造

```
入力トークン列
      ↓
  Embedding           ← トークンをベクトルに変換
      ↓
Positional Encoding   ← 位置情報を付加
      ↓
┌─────────────┐
│  Block × N  │       ← 同じ構造を N 層積み重ねる
│             │
│ LayerNorm   │
│ Attention   │       ← トークン間の関係を計算
│ LayerNorm   │
│ FFN         │       ← 各トークンを個別に変換
└─────────────┘
      ↓
   lm_head            ← 語彙サイズの確率分布に変換
      ↓
  次のトークン
```

### なぜ強いのか

| 要因 | 内容 |
|---|---|
| スケール則 | パラメータ・データ・計算量を増やすと性能が上がり続ける |
| 並列学習 | GPU で大規模データを高速に処理できる |
| 汎用性 | テキスト・画像・音声・コードなど何でも扱える |
| Transfer learning | 大規模事前学習 → タスク固有のファインチューニング |

### 現代への系譜

```
Transformer (2017)
    ↓
GPT系    ← デコーダのみ、自己回帰生成
BERT系   ← エンコーダのみ、双方向 Attention
T5系     ← エンコーダ・デコーダ、seq2seq
    ↓
GPT-4 / Claude / Gemini など
```

## 2. スタック型 Transformer

通常の Transformer は**同じ構造のブロックを縦に積み重ねる（stack）** 設計。

```
input
  ↓
Block 1  ← 同じ構造
  ↓
Block 2  ← 同じ構造
  ↓
 ...
  ↓
Block N  ← 同じ構造
  ↓
output
```

各ブロックは独立していて、**前のブロックの出力が次のブロックの入力**になるだけ。ブロック間で直接やり取りするキャッシュや状態はない。

### 各ブロックの中身（標準的な構成）

```
x → LayerNorm → Self-Attention → + (residual) → LayerNorm → FFN → + (residual) → x'
```

| 要素 | 役割 |
|---|---|
| LayerNorm | 安定化 |
| Self-Attention | トークン間の関係を学習 |
| FFN | 非線形変換・特徴抽出 |
| Residual | 勾配消失の防止 |

## 3. Self-Attention

Self-Attention は Transformer の**構成要素のひとつ**。

```
Transformer
  └── Block × N
        ├── Self-Attention   ← ここ
        └── FFN
```

### 用語の整理

| 用語 | 説明 |
|---|---|
| Attention | トークン間の関連度を計算する**仕組み** |
| Self-Attention | 同じ系列内のトークン同士で Attention をかける**操作** |
| Multi-Head Attention | Self-Attention を複数の「視点」で並列に行う**層** |
| Transformer | Self-Attention と FFN を積み重ねた**アーキテクチャ全体** |

### たとえると

```
Transformer    ＝  建物全体
Block          ＝  各フロア
Self-Attention ＝  フロア内の会議室（情報を集める場所）
FFN            ＝  フロア内の作業室（情報を処理する場所）
```

## 4. トークンの K・V（Key / Value）

各トークンは Attention 層に入ると **3つのベクトル** に線形変換される。

```
トークン x
  ├─ × W_Q → Q（Query）  "自分は何を探しているか"
  ├─ × W_K → K（Key）    "自分はどんな情報を持っているか"
  └─ × W_V → V（Value）  "実際に渡す情報の中身"
```

### 直感的なたとえ（図書館）

| | 役割 | たとえ |
|---|---|---|
| Q | 問い合わせ | 「猫について知りたい」 |
| K | 索引タグ | 「この本は猫の本です」 |
| V | 実際の内容 | 本の中身そのもの |

### 計算の流れ

```
1. 類似度スコア = Q × K^T           （どのトークンに注目するか）
2. 重み         = softmax(スコア / √d)  （正規化）
3. 出力         = 重み × V           （情報の加重平均）
```

### 具体例

```
"The cat sat on the mat"
       ↑
    このトークン (Q="sat") が他のトークンを参照する

K との類似度:
  "The"  → 0.1
  "cat"  → 0.8  ← 高い（主語として重要）
  "sat"  → 0.3
  "on"   → 0.2
  "the"  → 0.1
  "mat"  → 0.6  ← 高い（着地点として重要）

→ "cat" と "mat" の V を多めに混ぜて出力
```

### なぜ K と V を分けるのか

| もし K=V だったら | K と V を分けると |
|---|---|
| 「何を持っているか」と「渡す中身」が同じ | 索引と内容を独立して学習できる |
| 表現力が限定的 | より柔軟な情報抽出が可能 |

## 5. KV Cache

自己回帰生成（1トークンずつ生成）の際、**過去のトークンの K・V は毎ステップ同じ値**になる。これを毎回再計算するのは無駄なのでキャッシュする仕組み。

### なぜ必要か

| | キャッシュなし | キャッシュあり |
|---|---|---|
| 計算量 | O(T²) 毎ステップ | O(T) 増分のみ |
| メモリ | 少ない | K・V を保持する分増える |
| 速度 | 遅い | 速い |

### 仕組みのイメージ

```
ステップ1: "The" → K1, V1 を計算・保存
ステップ2: "cat" → K2, V2 を計算・保存、K1/V1 は再利用
ステップ3: "sat" → K3, V3 を計算・保存、K1/V1/K2/V2 は再利用
```

新しいトークンの Q だけを計算し、過去の K・V は保存済みのものを使う。Q をキャッシュしない理由は、Q は「今このステップで何を探すか」であり毎ステップ変わるから。

### メモリ使用量

```
メモリ量 = 2 × (layers) × (heads) × (head_dim) × (seq_len) × (batch) × dtype
```

長文・大バッチ・多層モデルになるほど急増する。対策として **PagedAttention**（vLLM）や **量子化 KV Cache** などが研究されている。

### 標準 KV Cache vs Depth KV Cache（MoDA）

| | 標準 KV Cache | Depth KV Cache |
|---|---|---|
| キャッシュの軸 | 時間（トークン） | 深さ（層） |
| 参照範囲 | 同一層の過去トークン | 他の層の K・V |
| 目的 | 推論高速化 | 層間情報共有 |

```
【標準】各層が自分の層内のトークン履歴をキャッシュ
Layer 1: [K1, K2, K3, ...]
Layer 2: [K1, K2, K3, ...]
Layer N: [K1, K2, K3, ...]

【Depth KV Cache】キャッシュが層をまたいで共有される
Block 1 → K, V を書き込む → [Layer 1 の K/V]
Block 2 → K, V を書き込む → [Layer 2 の K/V]  ← 他の層も参照できる
Block N →                 → [Layer N の K/V]
```

## 6. Prelude / Coda

音楽用語が転じたコードベースの命名慣習。

| | Prelude | Coda |
|---|---|---|
| 音楽的意味 | 前奏 | 終奏・締めくくり |
| コード上の役割 | 最初に読み込む共通定義 | 最後に実行する後処理・締め |
| 位置 | モデルの入り口側 | モデルの出口側 |

### Prelude

「デフォルトでインポートされる基本定義の集まり」。Rust の `std::prelude`、Haskell の `Prelude` モジュールが典型例。

Transformer のコードでは以下のようなものがまとめられることが多い：

- よく使う型エイリアス（`Tensor`、`Device` など）
- 共通のユーティリティ関数
- 設定・定数クラス
- よく使うサードパーティのシンボルの再エクスポート

### Coda

「後処理・後始末・締めくくり処理」をまとめたモジュールや関数群。

Transformer での例：

- 最終層の出力を受け取る後処理（RMSNorm → lm_head など）
- ログや損失の集計
- チェックポイント保存などの終了処理
